# Contributing to marker-groups.nvim

Thank you for your interest in contributing! This guide will help you get started.

## 🚀 Quick Start

1. **Fork** this repository
2. **Create a branch** using our naming conventions
3. **Make your changes** with clear commit messages
4. **Submit a pull request** with proper description
5. **Respond to review feedback**

## 🌿 Branch Naming

Use these standard prefixes for your branches:

```bash
feature/add-new-feature      # New features and enhancements
fix/resolve-memory-leak      # Bug fixes  
bugfix/header-styling        # Bug fixes (alternative)
hotfix/security-patch        # Critical production fixes
docs/update-readme           # Documentation updates
chore/bump-dependencies      # Maintenance tasks
refactor/simplify-parser     # Code refactoring
test/add-unit-tests          # Testing-related work
```

## 📝 Pull Request Guidelines

### PR Title Format
Use [Conventional Commits](https://conventionalcommits.org/) format:

```
type: description

Examples:
feat: add telescope integration
fix: resolve memory leak in marker cleanup  
docs: update installation instructions
chore: bump dependency versions
```

### PR Description Template
Please include this information in your PR description:

```markdown
## Description
Brief description of what this PR does.

## Type of Change
- [ ] Bug fix (non-breaking change which fixes an issue)
- [ ] New feature (non-breaking change which adds functionality)
- [ ] Breaking change (fix or feature that would cause existing functionality to not work as expected)
- [ ] Documentation update

## Related Issues
<!-- If this PR addresses any issues, list them here -->
Closes #123
Fixes #456

## Testing
- [ ] I have tested this change locally
- [ ] Existing tests pass
- [ ] I have added tests for new functionality

## Checklist
- [ ] My code follows the project's style guidelines
- [ ] I have performed a self-review of my code
- [ ] I have commented my code, particularly in hard-to-understand areas
- [ ] My changes generate no new warnings
```

## 🔗 Linking Issues

If your PR addresses existing issues:

1. **Reference issues in PR description** (not title)
2. Use GitHub keywords to auto-close issues when merged:
   - `Closes #123` - Closes the issue when PR is merged
   - `Fixes #456` - Fixes and closes the issue  
   - `Resolves #789` - Resolves and closes the issue

**Note**: The maintainer will review and confirm issue links during the PR review process.

## 💻 Commit Messages

Use conventional commit format:

```
type: description

Examples:
feat: add new authentication method
fix: resolve memory leak in cleanup
docs: update API documentation
test: add unit tests for parser
```

**No need to include issue numbers in commit messages** - handle issue linking in PR descriptions instead.

## 🧪 Testing

Before submitting your PR:

1. **Run tests** via `Makefile`:
   - `make test` – run all tests
   - `make test-unit` – unit tests only
   - `make test-integration` – integration tests only
   - `make test-file FILE=tests/unit/config_spec.lua` – run a single test file
2. **Check formatting**:
   - `make format` – format Lua code with StyLua
   - `make format-check` – verify formatting without changing files
3. **Lint (optional)**:
   - `make lint` – basic Lua syntax check; enhanced if `lua-language-server` is installed
4. **Verify functionality**: Test your changes locally with Neovim

## 📋 Review Process

1. **Automated checks** must pass (tests, linting, formatting)
2. **Maintainer review** - we'll review your code and may request changes
3. **Issue confirmation** - maintainer will confirm any issue links
4. **Merge** - once approved, your PR will be merged to main

## 🏷️ Releases

- Releases are created when the maintainer tags new versions
- Issues are automatically closed when related PRs are merged
- Release notes are auto-generated by GitHub Releases (no committed CHANGELOG.md)

## ❓ Questions?

- Start a discussion for questions or ideas
